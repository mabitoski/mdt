Production atelier

Comment fonctionnent les lots

Les lots actuels sont des lots de production. Ils servent a compter l'avancement atelier, pas a representer les palettes logistiques.

Automatique

Incrementation

Lorsqu'un technicien est assigne a un lot et qu'un poste remonte un report avec ce technicien, le lot est incremente automatiquement.

Manuel

Correction d'erreur

Si un PC n'a pas de lot ou a le mauvais lot, la fiche poste permet de lui en attribuer un, d'en changer un, ou de remettre "Aucun lot".

Regle serveur

Compteurs

Les compteurs sont recalcules cote serveur. Un lot en pause peut etre visible sur une machine, mais n'incremente pas le compteur produit.

  1. Creer le lot avec fournisseur, numero, objectif et priorite.
  2. Assigner les techniciens qui doivent alimenter ce lot.
  3. Laisser les postes remonter leurs rapports MDT normalement.
  4. Corriger manuellement un lot depuis la fiche poste en cas d'erreur de rattachement.

Attention

  • Un lot atelier n'est pas une palette logistique.
  • Le lot sert au suivi de production, pas au stockage ou a l'expedition.
  • Le report-zero reste le mode rapide. Pour une saisie manuelle complete, preferer le CSV normalise.

Saisie manuelle

Guide de normalisation CSV

Ce flux permet d'ajouter un ou plusieurs PC a la main avec le meme stockage que les remontees MDT: memes rapports, memes filtres, memes PDFs et meme fiche detail.

Modele CSV Import direct

Telecharger le modele

Le modele contient les en-tetes attendus. Une ligne correspond a un poste. Tu peux t'en servir pour un seul PC ou pour un import de plusieurs machines.

Telecharger le modele CSV

Laisse `lot_id` vide si l'affectation doit rester automatique.

Valeurs a respecter

  • `category`: `laptop`, `desktop`, `unknown`.
  • Statuts: `ok`, `nok`, `not_tested`, `absent`.
  • `bios_language`: `fr`, `en`, `not_tested`.
  • `bios_password`: `ok` = pas de mot de passe, `nok` = mot de passe present.
  • La MAC accepte `AA:BB:CC:DD:EE:FF` ou `AABBCCDDEEFF`.

Importer le CSV

Charge le fichier ici. Les lignes valides sont inserees dans les tables actives de l'application, pas dans un stockage a part.

Colonnes conseillees

  • Identifiants: `serial_number` ou `hostname` ou `mac_address`.
  • Atelier: `technician`, `category`, `vendor`, `model`, `os_version`.
  • Qualite: `camera_status`, `usb_status`, `keyboard_status`, `pad_status`, `badge_reader_status`, `bios_*`, `wifi_standard`, `*_test_status`.
  • Logistique optionnelle: `shipment_date`, `shipment_client`, `shipment_order_number`, `shipment_pallet_code`.

Checks effectifs

Ce qui est vraiment remonte par les scripts MDT

Cette section distingue les tests reelement automatiques, les tests dependants d'un outil annexe et les elements qui restent manuels ou heuristiques.

Automatique

Collecte technique

Le script remonte l'identite machine, l'inventaire materiel, les mesures WinSAT, le Wi-Fi, la batterie principale et une partie des peripheriques via WMI/CIM.

Semi-auto

Dependances externes

Certains checks ne sont fiables que si un outil existe vraiment sur le poste, par exemple le binaire camera ou la capture clavier.

Manuel

Non automatisable proprement

Les jumpers carte mere, le vrai etat de la pile BIOS ou certains champs BIOS ne sont pas exposes de facon generique et restent non fiables ou manuels.

Element Ce qui est fait aujourd'hui Mode Lecture operateur
Identite machine Hostname, serial, MAC, modele, vendor, OS, RAM, slots. Automatique Collecte standard WMI/CIM, fiable en usage courant.
Batterie principale Battery health et inventaire batterie principale Windows. Automatique Concerne la batterie du portable, pas la pile CMOS / BIOS.
Disques Inventaire disques + mesures WinSAT lecture / ecriture. Automatique + serveur Le serveur decide ensuite OK / NOK sur la performance disque.
CPU / RAM / GPU Scores et mesures WinSAT, avec fallbacks selon disponibilite. Automatique Le statut depend surtout de l'execution WinSAT et de la presence des mesures.
Reseau iperf si configure + ping rapide. Semi-auto Sans serveur iperf configure, le resultat peut rester not_tested.
Check disque logique chkdsk selon mode configure. Semi-auto Pas toujours lance, depend du mode et du contexte.
Diagnostic memoire Planification mdsched selon options. Semi-auto Ce n'est pas un resultat temps reel immediat dans la session courante.
Camera Presence PnP + test fonctionnel si un binaire camera est disponible. Semi-auto Sans exe camera, le statut peut rester not_tested meme si la camera existe.
USB Detection du controleur USB. Automatique Ce n'est pas un test port par port avec cle USB fonctionnelle.

Pave tactile Detection Windows + enrichissement possible par la capture clavier. Semi-auto / manuel En cas de detection floue, il faut garder un regard operateur.

Points de lecture importants

  • Sur desktop, les tests camera / USB / clavier / pave / badge sont volontairement sautes.
  • Le statut USB actuel correspond surtout a une presence de controleur, pas a un vrai test port.
  • Le statut clavier / pave devient pertinent quand la capture interactive a bien tourne.
  • La pile BIOS n'a pas de pourcentage universel exploitable via PowerShell classique.

Lecture des statuts

Comment interpreter OK, NOK, not_tested et absent

Le meme mot ne veut pas toujours dire la meme intensite de preuve. Il faut toujours le lire avec la nature du check.

OK

Le systeme considere le check satisfaisant. Pour un disque, c'est un verdict serveur sur la metrique. Pour un badge ou un USB, cela peut n'etre qu'une presence detectee.

NOK

Le check a explicitement echoue ou la valeur est sous seuil. Exemple actuel: disque trop lent cote serveur.

not_tested

Le test n'a pas pu etre mene, n'etait pas configure, ou est reserve a une saisie operateur. Ce n'est ni un OK ni un NOK.

absent

Le composant n'a pas ete detecte. C'est utile pour badge, camera ou peripheriques, mais ce n'est pas applicable a tous les checks.

Regles serveur deja actives

  • Lecture disque < 400 MB/s = NOK.
  • Ecriture disque < 350 MB/s = NOK.
  • Le serveur ecrase le verdict disque client et l'UI affiche ce verdict serveur.
  • Le serveur normalise aussi certaines informations BIOS / Wi-Fi lorsqu'elles existent.

Mode operatoire

Procedures types

Resume rapide de ce que doit faire chaque profil dans les scenarios les plus courants.

Atelier - passage standard

  1. Lancer le MDT avec le bon script.
  2. Attendre la remontee du report.
  3. Ouvrir la fiche poste si un doute existe.
  4. Verifier surtout les checks semi-auto ou manuels.
  5. Ajouter un commentaire si un point doit etre trace.

Atelier - correction d'un lot

  1. Ouvrir la fiche poste.
  2. Changer le selecteur Lot.
  3. Cliquer sur Appliquer.
  4. Verifier ensuite le libelle du lot et le compteur correspondant.

Qualite - analyse

  1. Utiliser Metrics pour la tendance globale.
  2. Utiliser Journal pour tracer une action precise.
  3. Utiliser Patchnotes pour connaitre les changements de logique.

Logistique - palettes ciblees

  1. Importer un CSV entree serial - palette.
  2. Importer un CSV sortie quand la palette quitte l'entrepot.
  3. Laisser le MDT associer automatiquement la palette par serial.

Ce flux est disponible via la page `/pallets`.

Ce qu'il faut remonter a la main aujourd'hui

  • Tout doute sur la langue BIOS ou le mot de passe BIOS.
  • Les anomalies clavier / pave si la capture interactive n'a pas tourne correctement.
  • Les cas ou la camera existe mais n'a pas pu etre testee fonctionnellement.
  • Tout sujet pile BIOS, jumpers ou comportement carte mere non expose par Windows.
